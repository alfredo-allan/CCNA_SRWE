1.1.1 Sequência de Inicialização do Switch

Antes de configurar um switch, você precisa ativá-lo e permitir que ele passe pela sequência de inicialização de cinco etapas. Este tópico aborda os conceitos básicos de configuração de um switch e inclui um laboratório no final.

Depois que um switch Cisco é ligado, ele passa pela seguinte sequência de inicialização em cinco etapas:

Etapa 1

Power-On Self-Test (POST)

Primeiro, o switch carrega um programa POST (Power-On Self-Test) armazenado na ROM. O POST verifica o subsistema da CPU. Ele testa a CPU, a DRAM e a parte do dispositivo flash que compõe o sistema de arquivos flash.

Etapa 2

Software Carregador de Inicialização

Depois, o switch carrega o software carregador de inicialização. O carregador de inicialização (boot loader) é um programa pequeno armazenado na ROM que é executado imediatamente após a conclusão bem-sucedida do POST.

Etapa 3

Inicialização de Baixo Nível da CPU

O carregador de inicialização executa a inicialização de baixo nível da CPU. Ele inicializa os registros da CPU, que controlam onde a memória física é mapeada, a quantidade de memória e sua velocidade.

Etapa 4

Inicialização do Sistema de Arquivos Flash

O carregador de inicialização (boot loader) inicializa o sistema de arquivos da memória flash na placa do sistema.

Etapa 5

Carregamento do Cisco IOS

Finalmente, o carregador de inicialização localiza e carrega uma imagem do software de sistema operacional Cisco IOS padrão na memória e passa o controle do switch para o IOS.

Dica de Monitoramento: Durante a inicialização, os LEDs do painel frontal do switch piscam sequencialmente. Se houver falha no POST, o LED do sistema (SYST) ficará âmbar (laranja), indicando que o hardware encontrou um problema crítico.

1.1.2 O Comando boot system

O switch tenta inicializar automaticamente usando informações na variável de ambiente BOOT. Se esta variável não estiver definida, o switch tentará carregar e executar o primeiro arquivo executável que encontrar. Nos Catalyst 2960 Series Switches, o arquivo de imagem normalmente está contido em um diretório que tem o mesmo nome do arquivo de imagem (excluindo a extensão de arquivo .bin).

O sistema operacional IOS inicializa as interfaces usando os comandos do Cisco IOS encontrados no arquivo startup-config. O arquivo startup-config é chamado config.text e está localizado em flash.

No exemplo, a variável de ambiente BOOT é definida usando o comando modo de configuração boot system global. Observe que o IOS está localizado em uma pasta distinta e o caminho da pasta é especificado. Use o comando show boot para ver como o arquivo de inicialização do IOS atual está definido.

Configuração da Variável de Inicialização:

S1(config)# boot system flash:/c2960-lanbasek9-mz.150-2.SE/c2960-lanbasek9-mz.150-2.SE.bin

Detalhamento dos Componentes do Comando:

Comando Definição
boot system O comando principal
flash: O dispositivo de armazenamento
c2960-lanbasek9-mz.150-2.SE/ O caminho para o sistema de arquivos
c2960-lanbasek9-mz.150-2.SE.bin O nome do arquivo IOS
Nota de Engenharia: Lembre-se sempre de salvar as alterações com o comando copy running-config startup-config para garantir que o switch aplique a nova variável BOOT na próxima inicialização.

1.1.3 Alternar Indicadores LED

Os switches Cisco Catalyst possuem várias luzes LED indicadoras de status. Você pode usar os LEDs do switch para monitorar rapidamente o desempenho e a atividade do dispositivo. Os switches de diferentes modelos e conjuntos de recursos terão LEDs diferentes, e sua colocação no painel frontal do switch também pode variar.

O botão Modo (7 na figura) é usado para alternar entre o status da porta, porta duplex, velocidade da porta e, se suportado, o status Power over Ethernet (PoE) dos LEDs da porta (8 na figura).

Esquema do Painel Frontal (Série Catalyst 2960):

+-------------------------------------------------------------------------+
|  CISCO                                      [ 1 ] [ 2 ] [ 3 ] [ 4 ]     |
|   (1) SYST  * [ 5 ] [ 6 ] [ 7 ] [ 8 ]     |
|   (2) RPS   * |
|   (3) STAT  * +---------------------------------------------+     |
|   (4) DUPLX * |  [Porta 1]  [Porta 3]  [Porta 5]  [Porta 7] |     |
|   (5) SPEED * |  [Porta 2]  [Porta 4]  [Porta 6]  [Porta 8] |     |
|   (6) PoE   * +---------------------------------------------+     |
|                     |===> (8) LEDs Indicadores de cada Porta      |     |
|  [MODE] (7)         +---------------------------------------------+     |
+-------------------------------------------------------------------------+
Pressione o botão Mode sucessivamente para mudar qual categoria de diagnóstico (de 1 a 6) os LEDs individuais das portas [8] estarão refletindo no momento.

Painel de Diagnóstico dos Indicadores LED:

1 LED de Sistema (SYST)

Mostra se o sistema está recebendo energia e está funcionando corretamente.

Status do LED Significado
Desligado Significa que o sistema não está ligado.
Verde O sistema está funcionando normalmente.
Âmbar (Laranja) O sistema está recebendo energia, mas não está funcionando corretamente.

2 LED do Sistema Redundante de Fonte (RPS)

Mostra o status do sistema de alimentação reserva (Redundant Power System).

Status do LED Significado
Desligado O RPS está desativado ou não foi conectado corretamente.
Verde O RPS está conectado e pronto para fornecer energia reserva.
Piscando em Verde O RPS está conectado mas não está disponível porque está fornecendo energia para outro dispositivo.
Amarelo O RPS está no modo de espera (standby) ou em uma condição de falha.
Piscando em Âmbar A fonte de alimentação interna do comutador (switch) falhou e o RPS está fornecendo energia de forma ativa.

3 LED de Status da Porta (STAT)

Indica que o modo de status da porta está selecionado quando o LED está verde. Este é o modo padrão do switch. Quando selecionado, os LEDs de porta exibirão:

Status do LED da Porta Significado
Desligado Não há link presente ou a porta foi desligada administrativamente (shutdown).
Verde Um link físico está presente (camada 1 ativa).
Piscando em Verde Há atividade de rede; a porta está enviando ou recebendo dados.
Alternando Verde/Âmbar Há uma falha de link detectada na porta.
Amarelo A porta está bloqueada pelo STP (Spanning Tree Protocol) para garantir que não exista nenhum loop no domínio de encaminhamento (normalmente as portas permanecem nesse estado por aprox. 30 segundos após a ativação).
Piscando em Âmbar A porta foi bloqueada ativamente para evitar um loop iminente no domínio de encaminhamento.

4 LED da Porta Duplex (DUPLX)

Indica que o modo duplex da porta está selecionado quando o LED principal está verde. Quando selecionado, os LEDs das portas indicam:

Status do LED da Porta Significado
Desligado A porta está operando em modo semi-duplex (Half-Duplex).
Verde A porta está operando em modo full-duplex.

5 LED da Porta SPEED

Indica que o modo de velocidade da porta está selecionado. Quando selecionado, os LEDs de porta exibirão a taxa de transmissão operacional:

Status do LED da Porta Significado
Desligado A porta está operando a uma velocidade de 10 Mbps.
Verde A porta está operando a uma velocidade de 100 Mbps.
Piscando em Verde A porta está operando na velocidade máxima de 1000 Mbps (1 Gbps).

6 LED do Modo Power over Ethernet (PoE)

Se o switch suportar fornecimento de energia por cabo de rede, este LED estará presente. Quando o modo PoE é selecionado pelo botão Mode, as cores indicam:

Status do LED Principal / Porta Significado
Desligado (Modo) O modo PoE não está selecionado; nenhuma porta teve energia negada ou entrou em falha.
Piscando em Âmbar (Modo) O modo PoE não está selecionado, mas pelo menos uma das portas teve energia negada ou apresenta falha de PoE.
Verde (Modo) O modo PoE está ativado e selecionado para exibição. Os LEDs das portas mudam de significado.
Desligado (Na Porta) O PoE está desligado/desativado nesta porta específica.
Verde (Na Porta) O PoE está ligado e fornecendo energia normalmente ao dispositivo conectado.
Alternando Verde/Âmbar (Porta) O PoE foi negado nesta porta porque o fornecimento de energia demandado pelo dispositivo conectado excede a capacidade de alimentação total do switch.
Piscando em Âmbar (Porta) O PoE está desligado nesta porta devido a uma condição de falha interna.
Âmbar Sólido (Na Porta) O fornecimento de PoE para esta porta foi desativado explicitamente.

1.1.4 Recuperando de um Travamento de Sistema

O carregador de inicialização (boot loader) fornece acesso emergencial ao switch caso o sistema operacional Cisco IOS não possa ser utilizado devido a arquivos de sistema ausentes, corrompidos ou danificados. O boot loader possui uma linha de comando (CLI) própria que permite gerenciar e interagir diretamente com os arquivos armazenados na memória flash, mesmo sem o IOS operacional.

Contexto de Recuperação: O ambiente do boot loader (prompt switch:) também é a ferramenta padrão para executar procedimentos de recuperação de senhas administrativas esquecidas ou perdidas.

Procedimento de Acesso via Console:

01

Conexão Física e Emulação

Conecte um PC utilizando o cabo de console à porta de console física do switch. Configure o software de emulação de terminal (como PuTTY ou Tera Term) com os parâmetros padrão de comunicação (9600 baud, 8 data bits, no parity, 1 stop bit, no flow control).

02

Desernegização

Desconecte o cabo de alimentação principal do switch para desligá-lo completamente.

03

Interrupção do Boot

Reconecte o cabo de alimentação ao switch. Dentro de uma janela máxima de 15 segundos, pressione e mantenha pressionado o botão Mode no painel frontal enquanto o LED do sistema (SYST) ainda estiver piscando em verde.

04

Liberação do Botão

Continue pressionando o botão Mode de forma contínua até que o LED do sistema fique brevemente âmbar e, em seguida, mude para verde sólido. Após essa transição visual, solte o botão Mode.

05

Prompt de Comando

O prompt de comando especial switch: será exibido na tela do software de emulação de terminal do PC, indicando o controle ativo do boot loader.

Interação e Comandos de Diagnóstico no Prompt switch:

Para visualizar a lista completa de comandos disponíveis no carregador de inicialização, você pode digitar help ou ?.

Por padrão, o switch busca as variáveis de boot. Para examinar o caminho configurado atualmente na variável de ambiente do sistema, execute o comando set. 
switch: set
BOOT=flash:/c2960-lanbasek9-mz.122-55.SE7/c2960-lanbasek9-mz.122-55.SE7.bin
(output omitted)

Antes de poder ler os arquivos do dispositivo de armazenamento, é obrigatório inicializar o sistema de arquivos flash através do comando flash_init:

Caso precise alterar a imagem de inicialização para mitigar um travamento ou carregar um novo IOS estável, redefina a variável de ambiente utilizando a sintaxe de atribuição direta BOOT=flash:nome_do_arquivo.bin. Valide o novo caminho com set e realize o boot manual invocando o comando boot sem argumentos:

1.1.5 Acesso ao Gerenciamento de Switches

Para preparar um switch para acesso de gerenciamento remoto, o dispositivo deve ser configurado com um endereço IP e uma máscara de sub-rede. Lembre-se de que, para gerenciar o switch a partir de uma rede remota, ele deve ser configurado adicionalmente com um gateway padrão. Esse procedimento é muito semelhante à configuração das informações de endereçamento IP em dispositivos hosts ordinários.

A SVI (Switch Virtual Interface) no switch precisa receber esse endereço IP operacional. A SVI é uma interface virtual de camada lógica, e não uma porta física do hardware do switch.

Nota de Implantação: Por padrão, a SVI do switch Catalyst vem mapeada de fábrica na VLAN 1. Um cabo de console é conectado a um PC host para que as configurações lógicas básicas iniciais da SVI possam ser injetadas pela primeira vez.

Esquema de Conexão e Topologia de Gerenciamento:

   +-------------------+
   |      [ PC1 ]      |
   +---------+---------+
             |
             | (Cabo do Console)
             |
   +---------v---------+                           +-------------------+
   |      [ S1 ]       |---------------------------|      [ R1 ]       |
   |  Switch Catalyst  |   (Link de Camada 2)      |     Roteador      |
   +-------------------+                           +---------+---------+
    SVI (VLAN 99)                                            |
    IP: 172.17.99.11/24                                      | Gateway Padrão:
    IPv6: 2001:db8:acad:99::11/64                            | 172.17.99.1
                                                             | 2001:db8:acad:99::1

Sintaxe CLI para Ativação da Interface SVI e Gateway (IPv4 e IPv6): 

S1# configure terminal
S1(config)# interface vlan 99
S1(config-if)# ip address 172.17.99.11 255.255.255.0
S1(config-if)# ipv6 address 2001:db8:acad:99::11/64
S1(config-if)# no shutdown
S1(config-if)# exit
S1(config)# ip default-gateway 172.17.99.1
S1(config)# end
S1# copy running-config startup-config

Parâmetros Fundamentais para Acesso Remoto:

Componente Finalidade e Mecânica Operacional
Endereço IP / Máscara Identifica logicamente o switch dentro da sub-rede local, permitindo o tráfego IP direto.
Switch Virtual Interface (SVI) Interface lógica associada a uma VLAN (ex: VLAN 1 ou VLAN 99) que atua como o host IP do switch.
Gateway Padrão (Default Gateway) Endereço IP do roteador local. Sem ele, o switch não consegue responder a conexões SSH/Telnet originadas de sub-redes remotas.
Cabo de Console Meio de acesso out-of-band essencial. Utilizado antes do switch possuir qualquer configuração IP funcional.
Diretriz de Segurança: Em ambientes de produção reais, evite utilizar a VLAN 1 nativa para o gerenciamento do switch. Recomenda-se criar uma VLAN dedicada e segura (como a VLAN 99 exemplificada no diagrama) para isolar o tráfego administrativo.

1.1.6 Exemplo de Configuração do SVI do Switch

Por padrão, o switch está configurado para ter seu gerenciamento controlado pela VLAN 1. Todas as portas são atribuídas à VLAN 1 por default. Por motivos de segurança, é uma prática recomendada usar uma VLAN diferente da VLAN 1 para a VLAN de gerenciamento, como a VLAN 99 no exemplo abaixo.

Observação: O SVI para a VLAN 99 não aparecerá como "up/up" até que a VLAN 99 seja criada e haja um dispositivo conectado a uma porta de switch associada à VLAN 99.

Passo 1: Configurar a Interface de Gerenciamento

No modo de configuração da interface VLAN, um endereço IPv4 e uma máscara de sub-rede são aplicados ao SVI de gerenciamento do switch.

Observação IPv6: O switch pode precisar ser configurado para IPv6. Por exemplo, antes de configurar o endereçamento IPv6 em um Cisco Catalyst 2960 executando o IOS versão 15.0, você precisará inserir o comando de configuração global sdm prefer dual-ipv4-and-ipv6 default e, em seguida, dar um reload no switch.
Tarefa Comandos IOS (Modo de Configuração)
Entre no modo de configuração global. 
S1# configure terminal
Entre no modo de configuração da interface para SVI. 
S1(config)# interface vlan 99
Configure o endereço IPv4 da interface de gerenciamento. 
S1(config-if)# ip address 172.17.99.11 255.255.255.0
Configure o endereço IPv6 da interface de gerenciamento. 
S1(config-if)# ipv6 address 2001:db8:acad:99::1/64
Ative a interface de gerenciamento. 
S1(config-if)# no shutdown
Volte para o modo EXEC privilegiado. 
S1(config-if)# end
Salve a configuração atual na configuração de inicialização. 
S1# copy running-config startup-config

Passo 2: Configurar o Gateway Padrão

O switch deve ser configurado com um gateway padrão se for gerenciado remotamente a partir de redes que não estão diretamente conectadas.

Observação de Gateway: Como ele receberá suas informações de gateway padrão de uma mensagem de anúncio de roteador (RA), o switch não requer um gateway padrão IPv6.
Tarefa Comandos IOS
Entre no modo de configuração global. 
S1# configure terminal
Configure o gateway padrão do switch. 
S1(config)# ip default-gateway 172.17.99.1
Volte para o modo EXEC privilegiado. 
S1(config)# end
Salve a configuração atual na configuração de inicialização. 
S1# copy running-config startup-config

Passo 3: Verificar Configuração

Os comandos show ip interface brief e show ipv6 interface brief são úteis para determinar o status das interfaces físicas e virtuais. A saída mostrada confirma que a interface VLAN 99 foi configurada com um endereço IPv4 e IPv6.

Aviso de Camada 3: Um endereço IP aplicado ao SVI é apenas para acesso de gerenciamento remoto ao switch; isso não permite que o switch roteie pacotes de Camada 3. 
S1# show ip interface brief
Interface    IP-Address       OK? Method Status                Protocol
Vlan99       172.17.99.11     YES manual down                  down
(output omitted)

S1# show ipv6 interface brief
Vlan99       [down/down]
    FE80::C27B:BCFF:FEC4:A9C1
    2001:DB8:ACAD:99::11
(output omitted)

1.2.1 Comunicação duplex

Configurar portas de switch

As portas de um switch podem ser configuradas de forma independente para diferentes necessidades. Este tópico aborda como configurar portas do switch, como verificar suas configurações, erros comuns e como solucionar problemas de configuração do switch.

Comunicação Full-Duplex

A comunicação full-duplex aumenta a eficiência da largura de banda, permitindo que ambas as extremidades de uma conexão transmitam e recebam dados simultaneamente. Isso também é conhecido como comunicação bidirecional e requer microssegmentação.

Uma LAN microssegmentada é criada quando uma porta do switch tem apenas um dispositivo conectado e está operando no modo full-duplex. Não há domínio de colisão associado a uma porta de switch operando no modo full-duplex.

Comunicação Half-Duplex

Ao contrário da comunicação full-duplex, a comunicação half-duplex é unidirecional. A comunicação semi-duplex cria problemas de desempenho porque os dados podem fluir em apenas uma direção por vez, geralmente resultando em colisões.

Hardware Legado: As conexões semi-duplex são normalmente vistas em hardware mais antigo, como hubs. A comunicação full-duplex substituiu o half-duplex na maioria dos hardwares modernos.

Requisitos Modernos de Infraestrutura

A evolução dos padrões de rede tornou a transmissão simultânea obrigatória para as tecnologias de alta velocidade atuais.

Alta Performance: A Ethernet Gigabit e as NICs de 10 Gb requerem conexões full-duplex para operar. No modo full-duplex, o circuito de detecção de colisão na NIC está desativado.

O full-duplex oferece 100% de eficiência em ambas as direções (transmissão e recepção). Isso resulta em uma duplicação do uso potencial da largura de banda declarada no link de rede.

1.2.2 Configurar as portas de switch na camada física

Configuração Padrão e Autonegociação

A configuração padrão para duplex e velocidade para portas de switch nos switches Cisco Catalyst 2960 e 3560 é automática. As portas 10/100/1000 operam no modo semi-duplex ou full-duplex quando estão definidas para 10 ou 100 Mbps e operam apenas no modo full-duplex quando está definido para 1000 Mbps (1 Gbps).

A negociação automática é útil quando as configurações de velocidade e duplex do dispositivo que se conecta à porta são desconhecidas ou podem ser alteradas. Ao conectar-se a dispositivos conhecidos, como servidores, estações de trabalho dedicadas ou dispositivos de rede, uma prática recomendada é definir manualmente as configurações de velocidade e duplex.

Aviso de Mismatch (Incompatibilidade): Configurações incompatíveis para o modo duplex e a velocidade das portas do switch podem causar problemas de conectividade. A falha de negociação automática cria configurações incompatíveis. Ao solucionar problemas de porta do switch, é mandatório que essas configurações sejam verificadas.

1.2.3 MDIX Automático

O que é o Auto-MDIX?

Até recentemente, certos tipos de cabo (direto ou cruzado) eram necessários ao conectar dispositivos. Conexões switch para switch ou switch para roteador necessitavam do uso de diferentes cabos Ethernet.

O uso do recurso de cruzamento automático da interface dependente do meio (auto-MDIX) em uma interface elimina esse problema. Quando o auto-MDIX está ativado, a interface detecta automaticamente o tipo de conexão de cabo necessário (direto ou cruzado) e configura a conexão adequadamente.

Cenário Sem Auto-MDIX: Ao se conectar a switches sem o recurso MDIX automático, os cabos straight-through (diretos) devem ser usados para conectar a dispositivos como servidores, estações de trabalho ou roteadores. Os cabos do tipo crossover (cruzados) devem ser usados para conectar a outros switches ou repetidores.

Requisitos para Funcionamento

Com o MDIX automático ativado, qualquer tipo de cabo pode ser usado para conectar a outros dispositivos e a interface se ajusta automaticamente para uma comunicação bem-sucedida.

Em switches Cisco mais recentes, o comando mdix auto no modo de configuração de interface habilita o recurso.

Regra Crítica do IOS: Ao usar o auto-MDIX em uma interface, a velocidade (speed) e o duplex (duplex) da interface devem ser configurados como auto para que o recurso funcione corretamente.

Habilitando o Auto-MDIX no Switch

O comando para habilitar o Auto-MDIX é emitido no modo de configuração da interface no switch, conforme mostrado abaixo: 

S1(config-if)# mdix auto

Observação de Hardware

Disponibilidade: O recurso Auto-MDIX é ativado por padrão nos switches Catalyst 2960 e Catalyst 3560, mas não está disponível nos switches Catalyst 2950 e Catalyst 3550 mais antigos.

Verificando o Status do Auto-MDIX

Para examinar a configuração Auto-MDIX para uma interface específica, use o comando show controllers ethernet-controller com a palavra-chave phy. Para limitar a saída às linhas que fazem referência ao Auto-MDIX, use o filtro | include Auto-MDIX. Como mostrado, a saída indica se está On ou Off para o recurso. 

S1# show controllers ethernet-controller fa0/1 phy | include MDIX
Auto-MDIX          : On   [AdminState=1  Flags=0x00052248]

1.2.6 Problemas da camada de acesso à rede

Diagnóstico com show interfaces

A saída do comando show interfaces é útil para detectar problemas comuns de mídia. Uma das partes mais importantes dessa saída é a exibição do status do protocolo de link de linha e dados, conforme mostrado no exemplo. 

S1# show interfaces fastEthernet 0/18
FastEthernet0/18 is up, line protocol is up (connected)
Hardware is Fast Ethernet, address is 0025.83e6.9092 (bia 0025.83e6.9092)MTU 1500 bytes, BW 100000 Kbit/s...

Entendendo os Parâmetros de Status

A primeira linha do comando traz dois indicadores cruciais para a análise do analista de redes:

  • Primeiro parâmetro (FastEthernet0/18 está ativo/up): Refere-se à camada de hardware e indica se a interface está recebendo um sinal de detecção de portadora.
  • Segundo parâmetro (line protocol is up): Refere-se à camada de link de dados e indica se os protocolos keepalive da camada de link de dados estão sendo recebidos.

Guia de Correção de Problemas

Com base na saída do comando show interfaces, possíveis problemas podem ser corrigidos da seguinte forma:

Interface Ativa / Protocolo Inativo (up/down): Se a interface estiver ativa e o protocolo de linha estiver inativo, existe um problema. Pode haver uma incompatibilidade de tipo de encapsulamento, a interface na outra extremidade pode estar desativada por erro ou pode haver um problema de hardware.
Ambos Desligados (down/down): Se o protocolo de linha e a interface estiverem ambos desligados, um cabo não está ligado ou existe algum outro problema de interface. Por exemplo, em uma conexão back-to-back, a outra extremidade da conexão pode estar administrativamente inativa.
Administrativamente Inativa (administratively down): Se a interface estiver administrativamente desativada, ela foi desativada manualmente (o comando shutdown foi emitido) na configuração ativa.

Próxima Etapa: Contadores de Erro

A saída do comando show interfaces também exibe contadores e estatísticas para a interface examinada, permitindo identificar erros de transmissão acumulados no meio físico.

1.2.7 Erros de entrada e saída de interface

O que são Erros de Entrada (Input Errors)?

"Erros de entrada" é a soma de todos os erros em datagramas recebidos na interface que está sendo examinada. Isso inclui contagem de runts, giants, CRC, no buffer, frame, overrun e ignored. Os erros de entrada relatados do comando show interfaces incluem o seguinte:

  • Frames Runt: Quadros Ethernet que são mais curtos do que o comprimento mínimo permitido de 64 bytes são chamados de runts. NICs com funcionamento inadequado normalmente são a causa de quadros runt em excesso, mas sua causa também se dá por colisões.
  • Giants: Os quadros Ethernet maiores que o tamanho máximo permitido são chamados gigantes.
  • CRC errors: Nas interfaces Ethernet e serial, os erros CRC geralmente indicam um erro de mídia ou cabo. As causas mais comuns incluem interferência elétrica, conexões soltas ou danificadas ou cabeamento incorreto. Se houver muitos erros de CRC, há muito ruído no link, e você deve verificar o cabo. Você também deve procurar as causas do ruído e eliminá-las.

O que são Erros de Saída (Output Errors)?

"Output errors" refere-se à soma de todos os erros que impediram a transmissão final de datagramas para fora da interface que está sendo examinada. Os erros de saída relatados do comando show interfaces incluem o seguinte:

  • Collisions: Colisões em operações half-duplex são normais. No entanto, você jamais verá colisões em uma interface configurada para a comunicação de duplex completo.
  • Late collisions: Uma colisão tardia refere-se a uma colisão que ocorre após a transmissão de 512 bits do quadro. Comprimentos excessivos de cabos são a causa mais comum de colisões tardias. Outra causa comum é a configuração incorreta de duplex. Por exemplo, você pode ter uma extremidade de uma conexão configurada para full-duplex e a outra para half-duplex. Você veria colisões tardias na interface configurada para half-duplex. Nesse caso, você deve configurar a mesma configuração duplex em ambas as extremidades. Uma rede corretamente projetada e configurada nunca deve ter colisões tardias.

Alguns erros de mídia não são graves o suficiente para fazer com que o circuito falhe, mas causam problemas de desempenho da rede. A tabela explica alguns desses erros comuns que podem ser detectados usando o comando show interfaces.

CRC Erros CRC são gerados quando a soma de verificação calculada não é a mesma que a soma de verificação recebida.
Erros de output Soma de todos os erros que impediram a transmissão final dos datagramas da interface que está sendo examinada.
Colisões Número de mensagens retransmitidas devido a uma colisão Ethernet.

1.2.8 Solucionando problemas da camada de acesso à rede

Cenário de Falhas na Infraestrutura

A maioria dos problemas que afetam uma rede comutada são encontrados durante a implementação original. Teoricamente, depois de instalada, uma rede continua a operar sem problemas. No entanto, o cabeamento fica danificado, as configurações mudam e os novos dispositivos são conectados ao switch que exigem alterações na configuração do switch. É necessária manutenção contínua e solução de problemas da infraestrutura de rede.

Para solucionar problemas de cenários que não envolvem conexão ou conexão ruim entre um comutador e outro dispositivo, siga o processo geral baseado no comando show interfaces.

Etapa A: Se a Interface Estiver Desativada (Down)

Caso o comando show interfaces indique que a porta não está ativa, siga os seguintes procedimentos de hardware e negociação:

1. Verificação Física de Cabos: Verifique se os cabos apropriados estão sendo usados. Além disso, verifique se há danos no cabo e nos conectores. Se houver suspeita de um cabo defeituoso ou incorreto, substitua o cabo imediatamente.
2. Incompatibilidade de Velocidade (Speed Mismatch): Se a interface ainda estiver inativa, o problema pode ser devido a uma incompatibilidade na configuração de velocidade. A velocidade de uma interface é normalmente negociada automaticamente; portanto, mesmo que seja aplicada manualmente a uma interface, a interface de conexão deve negociar automaticamente de acordo.

Se ocorrer incompatibilidade de velocidade em função de uma configuração incorreta ou de um problema de hardware ou software, isso pode causar a inatividade da interface. Defina manualmente a mesma velocidade em ambas as extremidades de conexão se houver suspeita de problema.

Etapa B: Se a Interface Estiver Ativa (Up) com Problemas

Se a interface estiver ativa, mas os problemas de conectividade ainda estiverem presentes, use os contadores estatísticos para diagnosticar falhas lógicas e elétricas:

1. Análise de Ruído Elétrico e EMI: Usando o comando show interfaces, verifique se há indicações de ruído excessivo. As indicações podem incluir um aumento nos contadores para erros de runts, gigantes (giants) e CRC. Se houver ruído excessivo, primeiro encontre e remova a fonte do ruído, se possível. Além disso, verifique se o cabo não excede o comprimento máximo permitido para a categoria e verifique o tipo de cabo usado.
2. Diagnóstico de Colisões e Duplex: Se o ruído não for um problema, verifique se há colisões excessivas. Se houver colisões ou colisões tardias (late collisions), verifique as configurações duplex em ambas as extremidades da conexão.

Assim como a configuração de velocidade, a configuração duplex geralmente é negociada automaticamente. Caso o problema não pareça ser de incompatibilidade do duplex, defina manualmente o duplex para full (completo) nas duas extremidades da conexão.

Resumo Estruturado do Processo

Seguindo a lógica do IOS, o fluxo final de troubleshooting resume-se em:

Condição da Porta Causa Provável Ação Corretiva Recomendada
Interface DOWN Cabo danificado, pinagem errada ou falha de Speed. Substituir cabo ou fixar velocidade manualmente nas duas pontas.
Interface UP (Com Runts/CRC) Interferência eletromagnética (EMI) ou cabo longo demais. Isolar fonte de ruído e validar comprimento máximo do meio.
Interface UP (Com Late Collisions) Incompatibilidade de Duplex (Half vs Full). Forçar configuração para duplex full em ambos os switches.

1.3.1 Operação Telnet

O que é e como funciona o Telnet?

Talvez você nem sempre tenha acesso direto ao switch quando precisar configurá-lo. Você precisa ser capaz de acessá-lo remotamente e é imperativo que seu acesso seja seguro. Este tópico aborda como configurar o Secure Shell (SSH) para acesso remoto. Uma atividade Packet Tracer lhe dá a oportunidade de experimentar isso sozinho.

Já o Telnet usa a porta 23. É um protocolo mais antigo que utiliza transmissão não segura de texto sem formatação da autenticação de login (nome de usuário e senha) e dos dados transmitidos entre os dispositivos de comunicação.

Riscos de Segurança (Texto Claro)

Por não utilizar criptografia, qualquer tráfego trafegando via Telnet pode ser facilmente interceptado e interpretado na rede.

Ameaça de Interceptação: Um ator ameaçador pode monitorar pacotes usando o Wireshark. Por exemplo, na análise de tráfego realizada, o ator de ameaça capturou com sucesso o nome de usuário admin e a senha ccna de uma sessão ativa do Telnet.

Captura de Tráfego: Wireshark (Follow TCP Stream)

Abaixo está a representação lúdica da reconstrução do fluxo TCP obtida pelo atacante ao farejar a porta 23:

Username: admin

S1> _

1.3.2 Operação SSH

O que é o Secure Shell (SSH)?

O Secure Shell (SSH) é um protocolo seguro que usa a porta 22. Ele fornece uma conexão de gerenciamento segura (criptografada) para um dispositivo remoto. O SSH deve substituir o Telnet para conexões de gerenciamento. O SSH fornece segurança para conexões remotas criptografando o tráfego de autenticação (nome de usuário e senha) e os dados transmitidos.

Proteção Wireshark: Ao contrário do Telnet, se um ator de ameaça interceptar o tráfego SSH com um analisador de protocolo (como o Wireshark), as informações coletadas serão ilegíveis por estarem completamente criptografadas.

Captura de Tráfego: Wireshark (Fluxo SSH Criptografado)

Uma captura de tela do Wireshark do tráfego SSH mostra dados ilegíveis que parecem "lixo eletrônico" para qualquer interceptador:

[Encrypted Packet Payload: Username and Password Unreadable]
