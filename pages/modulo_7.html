DHCPv4

Configuração, verificação e troubleshooting de atribuição dinâmica de endereçamento IP em redes corporativas.

7.0.1 Por que devo cursar este módulo?

Bem-vindo ao DHCPv4!

O protocolo DHCP (Dynamic Host Configuration Protocol) atribui dinamicamente endereços IP aos dispositivos. O DHCPv4 é projetado especificamente para uma infraestrutura baseada em IPv4. Isso significa que você, como administrador da rede, não precisa passar o dia configurando manualmente endereços IP estáticos para cada host presente no ambiente.

Em uma pequena residência ou escritório, a gerência estática não seria muito complexa, mas qualquer grande infraestrutura corporativa pode conter centenas ou mesmo milhares de dispositivos finais conectados simultaneamente.

INSIGHT TÉCNICO
Neste módulo, aprenderá a configurar um roteador Cisco IOS para atuar como um servidor DHCPv4 dedicado. Adicionalmente, verá como configurar o próprio roteador Cisco IOS para operar como um cliente DHCPv4 (recebendo IP em interfaces WAN, por exemplo).

7.1.1 Servidor e cliente DHCPv4

O Dynamic Host Configuration Protocol v4 (DHCPv4) atribui endereços IPv4 e outras informações de configuração de rede dinamicamente. Como os clientes desktop geralmente constituem a grande maioria dos nós de uma rede, o DHCPv4 é uma ferramenta extremamente útil para poupar o tempo dos administradores da rede.

Um servidor dedicado de DHCPv4 é escalável e relativamente fácil de gerenciar. Entretanto, em um pequeno escritório remoto ou local de escritórios domésticos, um roteador da Cisco pode ser configurado para proporcionar serviços DHCPv4 sem a necessidade de um servidor dedicado. O software Cisco IOS oferece suporte a um servidor DHCPv4 adicional, completo.

O servidor DHCPv4 atribui, ou aluga, dinamicamente, um endereço IPv4 de um pool de endereços por um período limitado de tempo escolhido pelo servidor ou até que o cliente não precise mais do endereço.

Os clientes alugam as informações do servidor por um período definido administrativamente. Os administradores configuram os servidores DHCPv4 para definir os aluguéis para expirarem em intervalos diferentes. O aluguel ocorre normalmente de 24 horas a uma semana ou mais. Quando o aluguel expira, o cliente precisa pedir outro endereço, embora geralmente receba novamente o mesmo endereço.

7.1.2 Operação do DHCPv4

O DHCPv4 funciona no modo cliente / servidor. Quando um cliente se comunica com um servidor DHCPv4, o servidor atribui ou arrenda o endereço IPv4 para esse cliente. O cliente se conecta à rede com esse endereço IPv4 concedido até que a concessão expire. O cliente deve entrar em contato com o servidor DHCP periodicamente para estender o aluguel.

Este mecanismo de aluguel garante que os clientes que se mudem ou fiquem sem energia não mantenham os endereços de que não precisam. Quando um aluguel expirar, o servidor DHCP devolverá o endereço ao pool, onde ele poderá ser realocado conforme necessário.

Ciclo de Vida do Aluguel (Lease Time)

1. Aluguel Inicial
O servidor DHCPv4 concede e vincula um endereço IPv4 disponível a um host cliente por um período temporário definido.
2. Renovação Periódica
O cliente ativo envia requisições periodicamente para estender a validade do seu aluguel atual antes do término do tempo limite.
3. Devolução ao Pool
Caso a concessão expire sem renovação, o endereço IPv4 é desalocado e retorna imediatamente para o pool de distribuição.

7.1.3 Etapas para obter um leasing

Quando o cliente inicializa (ou deseja ingressar em uma rede), inicia um processo de quatro etapas para obter uma concessão:

  1. 1. Descoberta do DHCP (DHCPDISCOVER)
  2. 2. Pacote de DHCP Offer (DHCPOFFER)
  3. 3. Solicitação de DHCP (DHCPREQUEST)
  4. 4. Reconhecimento de DHCP (DHCPACK)
INTERATIVIDADE
Clique em cada botão nas abas abaixo para acompanhar detalhadamente o processo de quatro etapas para obter uma concessão DHCPv4 e visualizar a sinalização correspondente.

Etapa 1. DHCP Discover (DHCPDISCOVER)

O cliente inicia o processo usando uma mensagem DHCPDISCOVER de broadcast com seu próprio endereço MAC para descobrir os servidores DHCPv4 disponíveis. Como o cliente não tem informações válidas de IPv4 durante a inicialização, ele usa endereços de broadcast de Camada 2 e Camada 3 para se comunicar com o servidor. O objetivo da mensagem DHCPDISCOVER é encontrar servidores DHCPv4 na rede.

Servidor DHCPv4
Cliente DHCPv4
1 DHCPDISCOVER (Broadcast)

Etapa 2. Oferta de DHCP (DHCPOFFER)

Quando o servidor DHCPv4 recebe uma mensagem DHCPDISCOVER, reserva o endereço IPv4 disponível para alugar para o cliente. O servidor também cria uma entrada ARP que consiste no endereço MAC do cliente solicitante e o endereço IPv4 alugado do cliente. O servidor DHCPv4 envia a mensagem DHCPOFFER de ligação ao cliente solicitante.

Servidor DHCPv4
Cliente DHCPv4
2 DHCPOFFER (Unicast / Broadcast)

Etapa 3. Solicitação de DHCP (DHCPREQUEST)

Quando o cliente recebe o DHCPOFFER do servidor, ele envia uma mensagem DHCPREQUEST. Esta mensagem é usada para para geração e renovação do aluguel. Quando usado para geração de aluguel, DHCPREQUEST atua como um aviso de aceitação de vinculação para o servidor selecionado para os parâmetros que oferecia e uma recusa implícita a todos os outros servidores que possam ter fornecido ao cliente uma oferta de vinculação.

Muitas redes corporativas usam vários servidores DHCPv4. A mensagem DHCPREQUEST é enviada na forma de um broadcast para informar esse servidor DHCPv4 e todos os outros servidores DHCPv4 sobre a oferta aceita.

Servidor DHCPv4
Cliente DHCPv4
3 DHCPREQUEST (Broadcast)

Etapa 4. Confirmação de DHCP (DHCPACK)

Ao receber a mensagem DHCPREQUEST, o servidor pode verificar as informações de concessão com um ping ICMP nesse endereço para garantir que não esteja sendo usado, criará uma nova entrada ARP para a concessão do cliente e responderá com uma mensagem DHCPACK. A mensagem DHCPACK é uma cópia de DHCPOFFER, exceto por uma mudança no campo do tipo de mensagem. Quando o cliente recebe a mensagem DHCPACK, ele registra as informações de configuração e pode executar uma pesquisa ARP para o endereço atribuído. Se não houver resposta ao ARP, o cliente sabe que o endereço IPv4 é válido e começa a usá-lo como seu próprio.

Servidor DHCPv4
Cliente DHCPv4
4 DHCPACK (Unicast)

7.1.4 Etapas para renovar um leasing

Antes da expiração da concessão, o cliente inicia um processo de duas etapas para renovar a concessão com o servidor DHCPv4, conforme detalhado a seguir:

  1. 1. Solicitação de DHCP (DHCPREQUEST):
    Antes que a concessão expire, o cliente envia uma mensagem DHCPREQUEST diretamente ao servidor DHCPv4 que originalmente ofereceu o endereço IPv4. Se um DHCPACK não for recebido dentro de um período especificado, o cliente envia outro DHCPREQUEST, de modo que um dos outros servidores DHCPv4 possa estender o aluguel.
  2. 2. Confirmação de DHCP (DHCPACK):
    Ao receber a mensagem DHCPREQUEST, o servidor verifica as informações de concessão retornando um DHCPACK.

Observação: Essas mensagens (principalmente o DHCPOFFER e o DHCPACK) podem ser enviadas como unicast ou transmitidas de acordo com a IETF RFC 2131.

2 DHCPACK (Unicast) — "Sua solicitação é reconhecida."

7.2.1 Etapas para configurar um servidor DHCPv4 do Cisco IOS

Um roteador Cisco rodando o software Cisco IOS pode ser configurado para atuar como um servidor DHCPv4 completo. O servidor DHCPv4 do Cisco IOS atribui e gerencia endereços IPv4 de pools de endereços especificados no roteador para os clientes DHCPv4.

Fluxo Lógico de Implementação do Servidor

Excluir Endereços
Remove IPs estáticos (Gateways, Switches, Servidores) do pool de atribuição automática.
Criar o Pool DHCP
Define um nome exclusivo para o pool e muda o contexto da CLI para o modo dhcp-config.
Configurar Parâmetros
Define a rede operacional (network), gateway padrão (default-router) e o servidor DNS.

Sintaxe de Comandos IOS Correspondente 

Router(config)# ip dhcp excluded-address 192.168.10.1 192.168.10.9
Router(config)# ip dhcp pool LAN-POOL-10
Router(dhcp-config)# network 192.168.10.0 255.255.255.0
Router(dhcp-config)# default-router 192.168.10.1
Router(dhcp-config)# dns-server 209.165.200.225

7.2.4 Desativar o Servidor DHCPv4 do Cisco IOS

O serviço DHCPv4 é habilitado por padrão em roteadores Cisco rodando o software Cisco IOS. Para desabilitar o serviço, use o comando de configuração global no service dhcp.

Use o comando de configuração global service dhcp para readmitir o servidor DHCPv4, conforme mostrado no exemplo.

Observação: Limpar as configurações do pool ou endereços excluídos não desativa o processo em si. Se o serviço não for explicitamente desativado, o roteador ainda poderá responder a requisições DHCPv4 vazias ou gerar overhead desnecessário de processamento.

Desativando e Reativando o Serviço DHCPv4 

R1(config)# no service dhcp
R1(config)# service dhcp

7.3.1 Configurar o Agente de Retransmissão DHCPv4

Em uma rede complexa, os servidores corporativos geralmente estão localizados em uma sub-rede centralizada. Esses servidores podem fornecer serviços DHCPv4, DNS, TFTP e FTP para a empresa. Os clientes da rede geralmente não estão na mesma sub-rede que esses servidores.

Como as mensagens de requisição DHCPv4 dos clientes são enviadas via Broadcast, os roteadores por padrão não encaminham essas transmissões de Camada 3 além da sub-rede local, bloqueando o tráfego.

Cenário de Encaminhamento de IP Helper

PC-A
DHCP Client
Camada 2 Broadcast
Switch S1
G0/0/0 (192.168.10.1)
Roteador R1
DHCP Relay Agent
G0/0/1: 10.1.1.1
Rede 10.1.1.0/30
Roteador R2
G0/0/0: 10.1.1.2
Rede 192.168.11.0/24
Servidor DHCPv4
192.168.11.5

Para resolver esse problema no roteador R1, você deve configurar a interface de entrada ligada aos clientes (neste exemplo, a interface GigabitEthernet 0/0/0) com o comando de configuração de interface ip helper-address [endereço-do-servidor].

Quando o roteador R1 recebe a transmissão de descoberta (DHCPDISCOVER) na interface G0/0/0, ele altera o endereço de origem para seu próprio endereço IP na interface de saída e converte o destino de broadcast para o endereço Unicast do servidor DHCPv4 (192.168.11.5), permitindo o roteamento entre as sub-redes.

Configuração do ip helper-address no Roteador R1 

R1(config)# interface gigabitethernet 0/0/0
R1(config-if)# ip helper-address 192.168.11.5
R1(config-if)# end
R1#

Outros Serviços Retransmitidos por Padrão

O comando ip helper-address encaminha por padrão não apenas requisições DHCPv4, mas também os seguintes 8 serviços baseados em UDP:

  • • Porta 37: Time
  • • Porta 49: TACACS
  • • Porta 53: DNS
  • • Porta 67: DHCP/BOOTP Server
  • • Porta 68: DHCP/BOOTP Client
  • • Porta 69: TFTP
  • • Porta 137: NetBIOS Name Service
  • • Porta 138: NetBIOS Datagram Service
