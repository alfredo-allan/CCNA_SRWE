10.1.1 Ataques de Rede hoje

A imprensa geralmente cobre ataques a redes corporativas. Basta pesquisar na Internet por "ataques de rede recentes" para encontrar informações atualizadas sobre ataques atuais. Muito provavelmente, esses ataques envolverão um ou mais dos seguintes itens:

DDoS

Negação de Serviço Distribuída - Ataque coordenado de muitos dispositivos, chamados zumbis, com a intenção de degradar ou interromper o acesso público ao site e aos recursos de uma organização.

Violação de Dados

Data Breach - Ataque no qual os servidores ou hosts de dados de uma organização são comprometidos para roubar informações confidenciais.

Malware

Ataque no qual os hosts de uma organização são infectados com software malicioso que causa vários problemas. Por exemplo, ransomware como o WannaCry, criptografa os dados em um host e bloqueia o acesso a eles até que um resgate seja pago.

10.1.2 Dispositivos para Segurança de Redes

Vários dispositivos de segurança de rede são necessários para proteger o perímetro da rede contra acesso externo. Esses dispositivos podem incluir um roteador habilitado para Rede Virtual Privada (VPN), um firewall de próxima geração (NGFW) e um dispositivo de Controle de Acesso à Rede (NAC).

Roteador habilitado para VPN

Clique para mais informações

Um roteador habilitado para VPN fornece uma conexão segura a usuários remotos através de uma rede pública e na rede corporativa. Os serviços de VPN podem ser integrados ao firewall.

IPsec SSL/TLS Remote Access

NGFW

Clique para mais informações

Um NGFW fornece inspeção de pacotes persistente, visibilidade e controle de aplicativos, um sistema de prevenção de intrusões (NGIPS) de última geração, proteção avançada contra malware (AMP) e filtragem de URL.

Deep Packet NGIPS AMP

NAC

Clique para mais informações

Um dispositivo NAC inclui serviços de autenticação, autorização e contabilização (AAA). Em empresas maiores, esses serviços podem ser incorporados a um dispositivo que pode gerenciar políticas de acesso em uma ampla variedade de usuários e tipos de dispositivos. O Cisco Identity Services Engine (ISE) é um exemplo de dispositivo NAC.

10.1.3 Proteção de Endpoints

Os dispositivos de LAN, como comutadores, controladores de rede local sem fio (WLCs) e outros dispositivos de ponto de acesso (AP) interconectam terminais. A maioria desses dispositivos é suscetível aos ataques relacionados à LAN que são abordados neste módulo.

Ameaças Internas

Mas muitos ataques também podem se originar de dentro da rede. Se um host interno estiver infiltrado, pode se tornar um ponto de partida para um agente de ameaças obter acesso a dispositivos críticos do sistema, como servidores e dados confidenciais.

O que são Endpoints?

Os endpoints são hosts que geralmente consistem em laptops, desktops, servidores e telefones IP, além de dispositivos próprios de funcionários que geralmente são chamados de traga seus próprios dispositivos (BYODs).

Laptops Desktops Servidores Telefones IP BYODs

Proteção de Endpoints

Os endpoints são particularmente suscetíveis a ataques relacionados a malware originados por e-mail ou navegação na web. Esses endpoints geralmente usam recursos de segurança tradicionais para hosts, como:

Antivírus/Antimalware Firewalls de Host HIPS

No entanto, hoje os endpoints são melhor protegidos por uma combinação de:

NAC AMP ESA WSA
Produtos para Proteção Avançada contra Malware (AMP) incluem soluções para endpoints, como o Cisco AMP para Endpoints.

10.2.1 Autenticação com uma senha local

No tópico anterior, você aprendeu que um dispositivo NAC fornece serviços AAA. Neste tópico, você aprenderá mais sobre o AAA e as maneiras de controlar o acesso.

Método mais simples (e menos seguro)

O método mais simples de autenticação de acesso remoto é configurar uma combinação de login e senha no console, linhas vty e portas auxiliares. Este método é o mais fácil de implementar, mas também é o mais fraco e menos seguro.

Este método não oferece responsabilidade e a senha é enviada em texto sem formatação. Qualquer pessoa com a senha pode obter acesso ao dispositivo.

Configuração Básica (Insegura) 

R1(config)# line vty 0 4
R1(config-line)# password ci5c0
R1(config-line)# login
% Senha configurada em texto plano - vulnerável a sniffing
R1(config-line)# exit
R1(config)#
Vulnerabilidade: A senha é transmitida em texto claro e pode ser capturada por sniffers de rede. Qualquer pessoa com acesso à senha pode se passar por qualquer usuário, sem rastreabilidade.

SSH - Método mais seguro

O SSH é uma forma mais segura de acesso remoto:

  • ✅ Requer um nome de usuário e uma senha, ambos criptografados durante a transmissão
  • ✅ O nome de usuário e a senha podem ser autenticados pelo método do banco de dados local
  • ✅ Oferece mais responsabilidade porque o nome de usuário é registrado quando um usuário faz login

Configuração SSH + Banco de Dados Local 

R1(config)# ip domain-name example.com
R1(config)# crypto key generate rsa general-keys modulus 2048
% Gerando chave RSA de 2048 bits...
% Chave RSA gerada com sucesso
R1(config)# username Admin secret Str0ng3rPa55w0rd
R1(config)# ip ssh version 2
R1(config)# line vty 0 4
R1(config-line)# transport input ssh
R1(config-line)# login local
% SSH configurado com sucesso - acesso criptografado
R1(config-line)# exit
R1(config)#
Seguro: O SSH utiliza criptografia para proteger as credenciais durante a transmissão e oferece rastreabilidade através do registro do nome de usuário.

Limitações do Banco de Dados Local

O método de banco de dados local possui algumas limitações:

Gerenciamento Distribuído

As contas de usuário devem ser configuradas localmente em cada dispositivo. Em um ambiente corporativo de grande porte com vários roteadores e switches para gerenciar, pode levar tempo para implementar e alterar os bancos de dados locais em cada dispositivo.

Sem Fallback

A configuração do banco de dados local não fornece método de autenticação reserva (fallback). Por exemplo, e se o administrador esquecer o nome de usuário e a senha para este dispositivo? Sem o método alternativo disponível para autenticação, a recuperação de senha é a única opção.

Solução: Servidor Centralizado

Uma solução melhor é fazer com que todos os dispositivos se refiram ao mesmo banco de dados de nomes de usuário e senhas de um servidor central.

10.2.2 Componentes AAA

AAA significa Autenticação, Autorização e Contabilização. O conceito AAA é semelhante ao uso de cartão de crédito:

  • ▸ O cartão de crédito identifica quem pode usá-lo
  • ▸ Quanto esse usuário pode gastar
  • ▸ Mantém uma conta de quais itens ou serviços o usuário comprou

Os Três Pilares do AAA

Autenticação

Quem é você?

Verifica a identidade do usuário que está tentando acessar a rede. Geralmente feito através de nome de usuário e senha, mas pode incluir biometria, tokens ou certificados digitais.

Login/Password Biometria Certificados

Autorização

Quanto você pode gastar?

Determina quais recursos o usuário autenticado pode acessar e quais ações ele pode executar. Controla privilégios como acesso a comandos, VLANs ou dispositivos específicos.

Níveis de Privilégio Políticas de Acesso VLANs

Contabilização

Em que você gastou?

Registra e audita todas as ações realizadas pelo usuário durante a sessão. Mantém um log completo de atividades, incluindo comandos executados, horários de acesso e recursos utilizados.

Auditoria Logs Relatórios

10.2.3 Autenticação AAA

Local e baseado em servidor são dois métodos comuns de implementação da autenticação AAA.

Autenticação AAA Local

O AAA local armazena nomes de usuário e senhas localmente em um dispositivo de rede, como o roteador Cisco. Os usuários se autenticam no banco de dados local. A AAA local é ideal para redes pequenas.

Com o método baseado em servidor, o roteador acessa um servidor AAA central. O servidor AAA contém os nomes de usuário e a senha para todos os usuários. O roteador usa os protocolos RADIUS ou TACACS+ para se comunicar com o servidor AAA. Quando existem vários roteadores e switches, o AAA baseado em servidor é mais apropriado.

Dica de Estudo:
  • RADIUS - Mais comum, usa UDP, criptografa apenas a senha
  • TACACS+ - Mais seguro, usa TCP, criptografa todo o pacote
  • Para redes com poucos dispositivos, use AAA Local
  • Para redes com muitos dispositivos, use Servidor AAA

10.2.4 Autorização AAA

A autorização AAA é automática e não requer que os usuários executem etapas adicionais após a autenticação. A autorização controla o que os usuários podem ou não fazer na rede após serem autenticados.

Como funciona a Autorização?

A autorização usa um conjunto de atributos que descrevem o acesso do usuário à rede. Esses atributos são usados pelo servidor AAA para determinar privilégios e restrições para esse usuário.

Dica de Estudo: A autorização é automática após a autenticação. Diferente da autenticação que pergunta "Quem é você?", a autorização pergunta "O que você pode fazer?".

10.2.6 802.1X

O padrão IEEE 802.1X é um protocolo de autenticação e controle de acesso baseado em porta. Este protocolo impede que estações de trabalho não autorizadas se conectem a uma LAN através de portas do switch acessíveis ao público.

Como funciona o 802.1X?

O servidor de autenticação autentica cada estação de trabalho conectada a uma porta de switch antes de disponibilizar qualquer serviço oferecido pelo switch ou pela LAN. Com a autenticação baseada em porta 802.1X, os dispositivos na rede têm funções específicas.

As Três Funções do 802.1X

Requerente

Cliente

É o dispositivo que deseja acessar a rede. Executa software cliente compatível com 802.1X, disponível para dispositivos com ou sem fio.

Laptop Smartphone Desktop

Autenticador

Switch / AP

Controla o acesso físico à rede com base no status de autenticação dos clientes. Atua como intermediário entre o cliente e o servidor de autenticação.

Switch AP Wireless

Autenticação

Servidor AAA

Valida a identidade do cliente e notifica o switch ou AP que o cliente está ou não autorizado a acessar a LAN e os serviços do switch.

10.4.1 Revisão da Operação de Switch

Neste tópico, o foco ainda está nos switches, especificamente em suas tabelas de endereços MAC e em como essas tabelas são vulneráveis a ataques.

Como funciona a Tabela de Endereços MAC?

10.4.2 Inundação da Tabela de Endereços MAC

Todas as tabelas MAC têm um tamanho fixo e, consequentemente, um switch pode ficar sem recursos para armazenar endereços MAC. Os ataques de inundação de endereço MAC aproveitam essa limitação bombardeando o switch com endereços MAC de origem falsos até que a tabela de endereços MAC do switch esteja cheia.

O que acontece quando a tabela MAC está cheia?

Quando isso ocorre, o switch trata o quadro como um unicast desconhecido e começa a inundar todo o tráfego de entrada em todas as portas na mesma VLAN sem fazer referência à tabela MAC. Agora, essa condição permite que um agente de ameaça capture todos os quadros enviados de um host para outro na LAN local ou na VLAN local.

Nota importante: O tráfego é inundado apenas dentro da LAN ou VLAN local. O agente de ameaças pode capturar apenas o tráfego na LAN ou VLAN local à qual está conectado.

A Ferramenta macof

A figura mostra como um agente de ameaça pode facilmente usar a ferramenta de ataque de rede macof para sobrecarregar uma tabela de endereços MAC.

macof é uma ferramenta de teste de segurança que gera rapidamente muitas fontes aleatórias e endereços IP de origem e de destino para sobrecarregar a tabela MAC do switch.

10.5.2 Ataques de Salto de VLAN

Um ataque de salto de VLAN permite que o tráfego de uma VLAN seja visto por outra VLAN sem a ajuda de um roteador.

Como funciona o Ataque de Salto de VLAN?

Em um ataque básico de salto de VLAN, o agente de ameaças configura um host para agir como um switch para aproveitar o recurso da porta de entroncamento automático ativada por padrão na maioria das portas de switches.

O agente de ameaça configura o host para falsificar a sinalização 802.1Q e o protocolo DTP (Dynamic Trunking Protocol) da Cisco para o tronco com o switch de conexão.

Consequências do Ataque

Se for bem-sucedido, o switch estabelece um link de tronco com o host. Agora, o agente de ameaças pode acessar todas as VLANs no switch.

Acesso a Todas as VLANs
O atacante pode ver tráfego de todas as VLANs

VLAN Hopping
Pula efetivamente entre VLANs sem roteador

Acesso a Servidores
O atacante obtém acesso à VLAN do servidor

Mitigação do Ataque de Salto de VLAN

Desabilitar DTP

Configure as portas como access em vez de trunk quando não forem necessárias. Use o comando switchport nonegotiate para desabilitar DTP.

VLANs Permitidas

Switch Spoofing

O atacante se passa por um switch legítimo e negocia um tronco com o switch real, obtendo acesso a todas as VLANs.

Double Tagging

O atacante envia quadros com duas tags VLAN (802.1Q) para contornar a VLAN nativa e acessar outras VLANs.

10.5.3 Ataque de Marcação Dupla de VLAN

Um agente de ameaça em situações específicas pode incorporar uma tag 802.1Q oculta no quadro que já possui uma tag 802.1Q. Essa tag permite que o quadro acesse uma VLAN que a tag 802.1Q original não especificou.

1 Envio do Quadro com Dupla Tag

O agente da ameaça envia um quadro 802.1Q com etiqueta dupla ao switch. O cabeçalho externo possui a tag VLAN do agente de ameaças, que é igual à VLAN nativa da porta de tronco. Para os fins deste exemplo, suponha que seja a VLAN 10. A etiqueta interna é a VLAN vítima, neste exemplo, VLAN 20.

Ethernet VLAN 10 VLAN 20 Dados
Atacante
Switch 1
Objetivo: Acessar a VLAN 20 (vítima) usando a VLAN 10 (nativa) como camuflagem.

2 Primeiro Switch - Remoção da Tag Externa

O quadro chega no primeiro switch, que analisa a primeira etiqueta 802.1Q de 4 bytes. O switch vê que o quadro é destinado à VLAN 10, que é a VLAN nativa. O switch encaminha o pacote para todas as portas da VLAN 10 após remover a etiqueta da VLAN 10. O quadro não é etiquetado novamente porque faz parte da VLAN nativa. Nesse ponto, a tag VLAN 20 ainda está intacta e não foi inspecionada pelo primeiro switch.

Switch 1
Remove VLAN 10
Intacta VLAN 20
Ethernet VLAN 20 Dados

3 Segundo Switch - Entrega na VLAN Alvo

O quadro chega ao segundo switch, que não sabe que deveria ser para a VLAN 10. O tráfego de VLAN nativo não é identificado pelo switch de envio, conforme especificado na especificação 802.1Q. O segundo switch examina apenas a tag 802.1Q interna inserida pelo agente de ameaças e vê que o quadro é destinado à VLAN 20, a VLAN de destino. O segundo switch envia o quadro para o destino ou o inunda, dependendo da existência de uma entrada existente na tabela de endereços MAC para o destino.

Switch 2
Entrega na VLAN 20
Servidor Alvo VLAN 20
Ethernet Dados

Características do Ataque

  • ▸ Unidirecional: O ataque funciona apenas em uma direção (do atacante para o alvo)
  • ▸ Requisito: O atacante deve estar na mesma VLAN nativa da porta de tronco
  • ▸ Objetivo: Contornar controles de acesso e alcançar VLANs restritas

Mitigação de Ataques de VLAN

Os ataques de salto e marcação dupla de VLAN podem ser evitados implementando as seguintes diretrizes de segurança de tronco:

  • ✅ Desative o entroncamento em todas as portas de acesso
  • ✅ Desative o entroncamento automático nos links de tronco para que os troncos sejam ativados manualmente
  • ✅ Certifique-se de que a VLAN nativa seja usada apenas para links de tronco

Configuração Segura para Prevenir Double Tagging 

=== CONFIGURAÇÃO SEGURA DE TRONCO ===

! Desabilitar DTP na porta de acesso
Switch(config)# interface FastEthernet 0/1
Switch(config-if)# switchport mode access
Switch(config-if)# switchport nonegotiate

! Configuração segura de porta trunk
Switch(config)# interface GigabitEthernet 0/24
Switch(config-if)# switchport mode trunk
Switch(config-if)# switchport nonegotiate
Switch(config-if)# switchport trunk native vlan 999
Switch(config-if)# switchport trunk allowed vlan 10,20,30

! Verificar configuração
Switch# show interfaces GigabitEthernet 0/24 switchport

Atenção! O ataque de marcação dupla de VLAN (Double Tagging) explora a VLAN nativa. Para prevenir, nunca use a VLAN 1 como VLAN nativa e desative DTP em todas as portas.

Os servidores DHCP fornecem dinamicamente informações de configuração de IP, incluindo endereço IP, máscara de sub-rede, gateway padrão, servidores DNS e muito mais para os clientes.

Detalhamento das Mensagens DHCP

DHCPDISCOVER

O cliente descobre servidores DHCP disponíveis na rede. Enviado como broadcast (255.255.255.255).

Origem: Cliente (0.0.0.0) → Destino: Broadcast (255.255.255.255)
DHCPOFFER

O servidor oferece um endereço IP disponível ao cliente. Pode ser unicast ou broadcast.

Origem: Servidor DHCP → Destino: Cliente ou Broadcast
DHCPREQUEST

O cliente solicita formalmente o endereço IP oferecido. Enviado como broadcast para informar todos os servidores.

Origem: Cliente → Destino: Broadcast (para todos os servidores)
DHCPACK

O servidor confirma a concessão do endereço IP. Pode ser unicast ou broadcast.

Origem: Servidor DHCP → Destino: Cliente ou Broadcast
